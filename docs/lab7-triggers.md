# Lab 7. ADF 트리거 (Schedule Trigger)

> **난이도:** ⭐⭐ 중급 | **소요시간:** 40분 | **사전 조건:** [Lab 1-A](lab1a-full-load.md) 완료

## 목표

- ADF 일정 트리거(Schedule Trigger)의 4가지 실전 패턴 실습
- Lab 1-A에서 작성한 `PL_Copy_DB2_to_ADLS_Full_HC` 파이프라인에 트리거 연결
- 트리거 생성 → 파이프라인 연결 → 게시(Publish) → 모니터링까지 전체 흐름

## 트리거 유형 개요

ADF는 3가지 트리거 유형을 제공합니다. 이번 Lab에서는 **일정 트리거**를 다룹니다.

| 유형 | 설명 | 사용 시나리오 |
|------|------|-------------|
| **일정(Schedule)** | 지정된 시간/간격으로 반복 실행 | 배치 ETL, 정기 수집 |
| 텀블링 윈도우(Tumbling Window) | 고정 크기 시간 창 단위 실행, 재처리 지원 | 시계열 데이터, 갭 없는 처리 |
| 이벤트(Event) | Blob 생성/삭제 이벤트 발생 시 실행 | 파일 도착 기반 처리 |

## 트리거 생성 공통 절차

```
[Author] → 파이프라인 선택 → 상단 [Add trigger] → [New/Edit]
→ [Choose trigger] → [+ New]
→ 트리거 설정 입력 → [OK]
→ 파라메터 매핑 (필요 시) → [OK]
→ ★ 반드시 [Publish all] 클릭 (게시하지 않으면 트리거 미동작)
```

> **주의:** 트리거는 **Publish(게시) 후에만 동작**합니다. Debug 모드에서는 트리거가 실행되지 않습니다.

## Lab 7-1. 5분마다 반복 실행

### 시나리오

실시간에 가까운 데이터 수집이 필요한 경우, 5분 간격으로 파이프라인을 자동 실행합니다.
개발/테스트 환경에서 트리거 동작을 빠르게 확인할 때도 유용합니다.

### 트리거 생성

```
[Author] → PL_Copy_DB2_to_ADLS_Full_HC 선택
→ [Add trigger] → [New/Edit] → [+ New]
```

| 설정 | 값 |
|------|-----|
| 이름 | `trigger_every_5min` |
| 설명 | 5분마다 전체 수집 실행 (테스트용) |
| 형식 | **일정** |
| 시작 날짜 | `4/13/2026, 1:18:33 AM` (현재 시간 이후) |
| 표준 시간대 | **Seoul (UTC+9)** |
| 되풀이 간격 | **5** |
| 되풀이 단위 | **분** |
| 종료 날짜 선택 | ☐ 체크 해제 (무한 반복) 또는 ☑ 체크 (테스트 시 종료일 지정) |
| 생성 시 트리거 시작 | ☐ 체크 해제 (Publish 후 수동 시작 권장) |

> **Tip:** 테스트용이라면 **종료 날짜를 설정**하세요. 5분마다 실행되면 비용이 누적됩니다.
>
> 예: 종료 날짜 = 시작 날짜 + 1시간 → 총 12회 실행 후 자동 종료

### [OK] → 파라메터 매핑

`PL_Copy_DB2_to_ADLS_Full_HC`는 파라메터가 없으므로 매핑 없이 [OK] 클릭.

### Publish

```
상단 [Publish all] 클릭 → 변경사항 확인 → [Publish]
```

### 실행 확인

```
[Monitor] → [Trigger runs] 탭
→ 5분 간격으로 트리거 실행 이력 확인
→ 각 실행의 Status, Start time 확인
```

### 트리거 중지

테스트 완료 후 반드시 트리거를 중지합니다:

```
[Manage] → [Triggers] → trigger_every_5min → [Deactivate]
→ [Publish all]
```

> **중요:** Deactivate 후에도 **Publish를 해야** 실제로 중지됩니다.

## Lab 7-2. 매시 정각 실행 (14:00, 15:00, 16:00 ...)

### 시나리오

업무 시간 중 매시간 데이터를 수집합니다. 예: 오후 2시부터 저녁 6시까지 매시 정각 실행.

### 트리거 생성

```
[Author] → PL_Copy_DB2_to_ADLS_Full_HC 선택
→ [Add trigger] → [New/Edit] → [+ New]
```

| 설정 | 값 |
|------|-----|
| 이름 | `trigger_hourly` |
| 설명 | 매시 정각 수집 (업무시간) |
| 형식 | **일정** |
| 시작 날짜 | `4/14/2026, 2:00:00 PM` (다음 실행 시작점) |
| 표준 시간대 | **Seoul (UTC+9)** |
| 되풀이 간격 | **1** |
| 되풀이 단위 | **시간** |
| 종료 날짜 선택 | ☐ (무한 반복) |

### 실행 시간 예시

```
14:00 KST → 실행
15:00 KST → 실행
16:00 KST → 실행
17:00 KST → 실행
18:00 KST → 실행
  ...      → 24시간 반복
```

> **업무 시간만 실행하고 싶다면?**
>
> 일정 트리거의 단순 되풀이로는 "14시~18시만 실행"을 직접 설정할 수 없습니다.
> 이 경우 두 가지 방법이 있습니다:
>
> **방법 1: 텀블링 윈도우 트리거 사용**
> - 텀블링 윈도우는 시작/종료 시간 내에서만 실행
>
> **방법 2: 파이프라인 내에서 시간 체크**
> - 파이프라인 첫 Activity에서 현재 시간을 체크하여 업무 시간 외면 Skip:
> ```
> @if(
>     and(
>         greaterOrEquals(int(formatDateTime(convertTimeZone(utcNow(),'UTC','Korea Standard Time'),'HH')), 14),
>         lessOrEquals(int(formatDateTime(convertTimeZone(utcNow(),'UTC','Korea Standard Time'),'HH')), 18)
>     ),
>     true,
>     false
> )
> ```

### Publish & Monitor

```
[Publish all] → [Monitor] → [Trigger runs] 탭에서 매시 실행 확인
```

## Lab 7-3. 매주 일요일 새벽 실행

### 시나리오

주간 배치 작업으로, 매주 일요일 새벽 2시에 전체 데이터를 수집합니다.
시스템 부하가 적은 시간대에 Full Load를 수행하는 전형적인 패턴입니다.

### 트리거 생성

```
[Author] → PL_Copy_DB2_to_ADLS_Full_HC 선택
→ [Add trigger] → [New/Edit] → [+ New]
```

| 설정 | 값 |
|------|-----|
| 이름 | `trigger_weekly_sunday` |
| 설명 | 매주 일요일 새벽 2시 전체 수집 |
| 형식 | **일정** |
| 시작 날짜 | `4/19/2026, 2:00:00 AM` (가장 가까운 일요일) |
| 표준 시간대 | **Seoul (UTC+9)** |
| 되풀이 간격 | **1** |
| 되풀이 단위 | **주** |

되풀이 단위를 **"주"**로 선택하면 **고급 되풀이 옵션**이 나타납니다:

### 고급 되풀이 옵션

| 설정 | 값 |
|------|-----|
| **다음 요일에 실행** | **일** (일요일 선택) |
| **다음 시간에 실행 — 시간** | **2** |
| **다음 시간에 실행 — 분** | **0** |

> 스크린샷처럼 요일 버튼(일/월/화/수/목/금/토)이 표시됩니다.
> **"일"** 버튼을 클릭하여 활성화합니다.

### 실행 시간 예시

```
2026-04-19 (일) 02:00 KST → 실행
2026-04-26 (일) 02:00 KST → 실행
2026-05-03 (일) 02:00 KST → 실행
  ...
```

### 복수 요일 실행

여러 요일에 실행하려면 요일 버튼을 복수 선택합니다:

| 시나리오 | 요일 선택 | 설명 |
|---------|----------|------|
| 주중 매일 | 월, 화, 수, 목, 금 | 평일 배치 |
| 주 2회 | 수, 일 | 수요일 + 일요일 |
| 주말만 | 토, 일 | 주말 배치 |

### Publish & Monitor

```
[Publish all] → [Monitor] → [Trigger runs]
→ 다음 일요일 02:00에 실행 이력 확인
```

## Lab 7-4. 매월 마지막 날 실행

### 시나리오

월말 마감 배치로, 매월 마지막 날 새벽 3시에 전체 수집을 수행합니다.
월말 정산, 스냅샷 저장 등에 사용됩니다.

> **주의:** ADF 일정 트리거에는 "매월 마지막 날"을 직접 지정하는 옵션이 없습니다.
> 따라서 **두 가지 대안** 중 하나를 사용합니다.

### 방법 A: 매일 실행 + 파이프라인 내 날짜 체크 (권장)

매일 트리거를 실행하되, 파이프라인 내에서 "오늘이 월 마지막 날인지" 체크하여
마지막 날에만 실제 작업을 수행합니다.

#### 트리거 생성

| 설정 | 값 |
|------|-----|
| 이름 | `trigger_daily_for_month_end` |
| 설명 | 매일 새벽 3시 실행 (월말만 실제 수행) |
| 형식 | **일정** |
| 시작 날짜 | `4/14/2026, 3:00:00 AM` |
| 표준 시간대 | **Seoul (UTC+9)** |
| 되풀이 간격 | **1** |
| 되풀이 단위 | **일** |

#### 파이프라인에 월말 체크 Activity 추가

파이프라인 맨 앞에 **If Condition Activity**를 추가합니다:

**Activity 이름:** `If_Last_Day_Of_Month`

**Expression:**
```
@equals(
    formatDateTime(addDays(
        startOfDay(convertTimeZone(utcNow(), 'UTC', 'Korea Standard Time')),
        1
    ), 'dd'),
    '01'
)
```

| 로직 | 설명 |
|------|------|
| `convertTimeZone(utcNow(), 'UTC', 'Korea Standard Time')` | 현재 시간을 한국시간으로 변환 |
| `startOfDay(...)` | 당일 00:00:00 |
| `addDays(..., 1)` | 내일 날짜 |
| `formatDateTime(..., 'dd')` | 내일의 일(day) 추출 |
| `equals(..., '01')` | 내일이 1일이면 = 오늘이 월 마지막 날 |

**True 분기:** Copy Activity (실제 작업 수행)

**False 분기:** 비워둠 (아무 것도 안 함 — 파이프라인 바로 종료)

```
파이프라인 흐름:

┌─────────────────┐          ┌──────────────────┐
│ If_Last_Day     │──True──▶│ Copy_DB2_to_ADLS │
│ _Of_Month       │          │ (실제 작업)       │
│                 │          └──────────────────┘
│ 내일 = 1일?     │
│                 │──False──▶ (종료, 아무 것도 안 함)
└─────────────────┘
```

#### 실행 예시

```
2026-04-28 03:00 → If 체크 → 내일=29일 → False → Skip
2026-04-29 03:00 → If 체크 → 내일=30일 → False → Skip
2026-04-30 03:00 → If 체크 → 내일=01일 → True  → ✅ Copy 실행!
2026-05-01 03:00 → If 체크 → 내일=02일 → False → Skip
  ...
2026-05-31 03:00 → If 체크 → 내일=01일 → True  → ✅ Copy 실행!
2026-06-01 03:00 → If 체크 → 내일=02일 → False → Skip
  ...
2026-02-28 03:00 → If 체크 → 내일=01일 → True  → ✅ Copy 실행! (윤년 자동 처리)
```

> **장점:** 2월(28일/29일), 4월(30일), 7월(31일) 등 월별 마지막 날이 달라도 정확하게 동작합니다.

### 방법 B: 매월 28일 고정 실행 (간단 대안)

모든 월에 28일이 존재하므로, 28일 고정으로 실행합니다.
정확한 월말이 아니어도 되는 경우에 사용합니다.

#### 트리거 생성

| 설정 | 값 |
|------|-----|
| 이름 | `trigger_monthly_28th` |
| 설명 | 매월 28일 새벽 3시 |
| 형식 | **일정** |
| 시작 날짜 | `4/28/2026, 3:00:00 AM` |
| 표준 시간대 | **Seoul (UTC+9)** |
| 되풀이 간격 | **1** |
| 되풀이 단위 | **개월** |

되풀이 단위를 **"개월"**로 선택하면 고급 옵션이 나타납니다:

#### 고급 되풀이 옵션 (월별)

| 설정 | 값 |
|------|-----|
| **다음 날짜에 실행** | **28** (매월 28일) |
| **다음 시간에 실행 — 시간** | **3** |
| **다음 시간에 실행 — 분** | **0** |

> **한계:** 실제 월말(30일, 31일)이 아닌 28일에 실행되므로, 28일~말일 사이 데이터가 다음 수집에 포함됩니다. 정확한 월말이 필요하면 **방법 A**를 사용하세요.

## 트리거 관리

### 트리거 목록 확인

```
[Manage] → [Triggers]
→ 전체 트리거 목록, 상태(Started/Stopped), 마지막 실행 시간 확인
```

### 트리거 활성화 / 비활성화

```
[Manage] → [Triggers] → 트리거 선택 → [Activate] 또는 [Deactivate]
→ ★ [Publish all] 클릭 필수
```

| 상태 | 설명 |
|------|------|
| **Started** | 트리거 활성 → 스케줄에 따라 자동 실행 |
| **Stopped** | 트리거 비활성 → 실행하지 않음 |

### 트리거 삭제

```
[Manage] → [Triggers] → 트리거 선택 → [Delete]
→ [Publish all]
```

> 트리거 삭제 전 반드시 **Deactivate**해야 합니다. 활성 상태에서는 삭제 불가.

### 트리거 실행 이력 확인

```
[Monitor] → [Trigger runs] 탭
→ 시간 범위 필터 설정
→ Status: Succeeded / Failed / In Progress
→ 각 트리거 실행 → 연결된 파이프라인 실행 추적 가능
```

## 트리거 + 파라메터화 파이프라인 연동

Lab 6의 `PL_Master_Ingestion`처럼 파라메터가 있는 파이프라인에 트리거를 연결하면,
트리거 생성 시 **파라메터 값을 지정**할 수 있습니다:

```
[Add trigger] → [New/Edit] → [+ New] → 트리거 설정 → [OK]
→ ★ Pipeline parameters 화면이 나타남:

  p_source_system : DB2_ONPREM
  p_load_date     : (비워둠 → 파이프라인 내 Set_Load_Date에서 자동 생성)
```

| 방법 | 설명 |
|------|------|
| 파라메터 비워둠 | 파이프라인 Default Value 또는 내부 로직 사용 |
| 값 직접 입력 | 트리거마다 다른 값 전달 가능 |
| 시스템 변수 사용 | `@trigger().scheduledTime` — 트리거 예정 시간 |

### 트리거 시스템 변수

트리거 파라메터 매핑에서 사용할 수 있는 시스템 변수:

| 변수 | 설명 | 예시 값 |
|------|------|--------|
| `@trigger().scheduledTime` | 트리거 예정 시간 (UTC) | `2026-04-13T17:00:00Z` |
| `@trigger().startTime` | 트리거 실제 시작 시간 | `2026-04-13T17:00:03Z` |

**활용 예시 — p_load_date에 트리거 시간 기반 날짜 전달:**

```
@formatDateTime(
    convertTimeZone(trigger().scheduledTime, 'UTC', 'Korea Standard Time'),
    'yyyyMMdd'
)
```

| **Publish 필수** | 트리거 생성/변경/삭제 후 반드시 Publish — 미게시 시 동작 안 함 |
| **시간대** | Seoul (UTC+9) 설정 필수 — UTC 기본값이면 한국시간과 9시간 차이 |
| **동시 실행** | 같은 파이프라인에 여러 트리거 연결 가능 — 동시 실행 시 주의 |
| **비용** | 트리거 실행 횟수 × Activity 실행 비용 — 테스트 트리거는 반드시 중지 |
| **최소 간격** | 일정 트리거 최소 되풀이 간격 = **1분** |
| **실패 재실행** | 일정 트리거는 실패 시 자동 재실행 안 함 — 텀블링 윈도우는 재처리 지원 |
| **파이프라인 실행 중 충돌** | 이전 실행이 아직 진행 중이면 새 트리거 실행은 **별도 인스턴스**로 시작 (동시 실행) |
